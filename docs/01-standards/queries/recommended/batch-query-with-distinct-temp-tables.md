# QRY-REC-001. Разделение поиска ссылок в пакетном запросе через ВЫБРАТЬ РАЗЛИЧНЫЕ и временные таблицы

## Статус
Проверено

## Уровень
Рекомендуемая практика

## Область
Запросы

## Краткое правило
Если пакетный запрос соединяет входные данные с несколькими большими таблицами БД для получения ссылок по натуральным ключам — выноси каждый поиск в отдельный пакет с `ВЫБРАТЬ РАЗЛИЧНЫЕ ... ПОМЕСТИТЬ втXxx`, а затем собирай результат через JOIN на временные таблицы.

## Зачем это нужно
Один запрос с несколькими LEFT JOIN на большие таблицы (справочники, документы) по входным данным:
- Создаёт тяжёлый план выполнения (декартово произведение промежуточных результатов)
- Может давать неожиданные дубли при множественных совпадениях на разных этапах JOIN
- Затрудняет диагностику: непонятно, на каком этапе потерялись или задублировались записи

Разделение на этапы:
- Снижает нагрузку: JOIN с таблицей БД идёт по малому набору уникальных ключей
- Упрощает диагностику: каждая временная таблица — это контрольная точка
- Позволяет вставить диагностические пакеты между этапами (ненайденные записи, дубли)

## Как применять
1. Определи, по каким натуральным ключам нужно найти ссылки (артикул → номенклатура, номер+дата → документ и т.п.).
2. Для каждого поиска создай отдельный пакет: `ВЫБРАТЬ РАЗЛИЧНЫЕ` уникальные ключи из входных данных → `LEFT JOIN` на таблицу БД → `ПОМЕСТИТЬ` во временную таблицу.
3. В финальном пакете собери результат через JOIN на временные таблицы (без повторного обращения к таблицам БД).
4. Добавь диагностические пакеты: `ВЫБРАТЬ ... ИЗ втXxx ГДЕ Ссылка ЕСТЬ NULL` — для ненайденных записей.
5. Добавь пакет проверки дублей: `СГРУППИРОВАТЬ ПО ... ИМЕЮЩИЕ КОЛИЧЕСТВО(РАЗЛИЧНЫЕ Ссылка) > 1`.

## Как проверять в review
- Если в запросе больше одного LEFT JOIN на таблицу БД по натуральному ключу — проверяй, стоит ли разделить.
- Проверяй наличие `ВЫБРАТЬ РАЗЛИЧНЫЕ` в промежуточных пакетах (без РАЗЛИЧНЫЕ смысл оптимизации теряется).
- Проверяй наличие диагностических пакетов для ненайденных записей.

## Пример правильно
```bsl
// Пакет 0: входные данные
"ВЫБРАТЬ ... ПОМЕСТИТЬ втДанные ИЗ &ДанныеExcel КАК Источник
;
// Пакет 1: уникальные артикулы -> номенклатура (малый набор)
ВЫБРАТЬ РАЗЛИЧНЫЕ
    втДанные.НоменклатураАртикул,
    Ном.Ссылка КАК НоменклатураСсылка
ПОМЕСТИТЬ втНоменклатура
ИЗ втДанные
    ЛЕВОЕ СОЕДИНЕНИЕ Справочник.Номенклатура КАК Ном
    ПО втДанные.НоменклатураАртикул = Ном.Артикул
;
// Пакет 2: уникальные номер+дата -> документы (малый набор)
ВЫБРАТЬ РАЗЛИЧНЫЕ
    втДанные.Номер, втДанные.Дата,
    Док.Ссылка КАК ДокументСсылка
ПОМЕСТИТЬ втДокументы
ИЗ втДанные
    ЛЕВОЕ СОЕДИНЕНИЕ Документ.РеализацияТоваровУслуг КАК Док
    ПО втДанные.Номер = Док.Номер И втДанные.Дата = Док.Дата
;
// Пакет 3: сборка из временных таблиц (без JOIN на БД)
ВЫБРАТЬ ...
ИЗ втДанные
    ЛЕВОЕ СОЕДИНЕНИЕ втНоменклатура ПО ...
    ЛЕВОЕ СОЕДИНЕНИЕ втДокументы ПО ...
;
// Пакет 4: диагностика - ненайденная номенклатура
ВЫБРАТЬ РАЗЛИЧНЫЕ НоменклатураАртикул ИЗ втНоменклатура ГДЕ НоменклатураСсылка ЕСТЬ NULL
;
// Пакет 5: диагностика - ненайденные документы
ВЫБРАТЬ РАЗЛИЧНЫЕ Номер, Дата ИЗ втДокументы ГДЕ ДокументСсылка ЕСТЬ NULL"
```

## Пример неправильно
```bsl
// Один тяжёлый запрос с тремя JOIN на таблицы БД
"ВЫБРАТЬ
    Ном.Ссылка, Хар.Ссылка, Док.Ссылка
ИЗ &ДанныеExcel КАК Данные
    ЛЕВОЕ СОЕДИНЕНИЕ Справочник.Номенклатура КАК Ном
        ПО Данные.Артикул = Ном.Артикул
    ЛЕВОЕ СОЕДИНЕНИЕ Справочник.ХарактеристикиНоменклатуры КАК Хар
        ПО Данные.Характеристика = Хар.Наименование
    ЛЕВОЕ СОЕДИНЕНИЕ Документ.РеализацияТоваровУслуг КАК Док
        ПО Данные.Номер = Док.Номер И Данные.Дата = Док.Дата"
// Проблемы: тяжёлый план, дубли при множественных совпадениях, нет диагностики
```

## Исключения
Если входных данных мало (единицы записей) и JOIN идёт на одну таблицу — разделение избыточно.

## Источники
- Практический опыт пакетной загрузки данных из файлов в документы 1С
